## 一、AI产品生命周期与产品经理职责

### AI产品生命周期（4大阶段）

| 阶段 | 说明 |
|------|------|
| **阶段1** | 需求分析 |
| **阶段2** | 产品设计 |
| **阶段3** | 开发与训练 |
| **阶段4** | 上线与迭代 |

### 产品经理在各阶段的角色

| 阶段 | 产品经理职责 |
|------|-------------|
| **需求分析** | 理解业务痛点、确定AI能力边界、分析可行性 |
| **产品设计** | 功能设计、交互设计、输出PRD |
| **开发与训练** | 协调技术团队、跟踪进度、效果评估 |
| **上线与迭代** | 数据分析、用户反馈、优化迭代 |

## 三、AI产品设计模式

### 发散式交互 vs 约束式交互

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **发散式交互** | 让用户像"自由对话"一样和AI交流，输入不受限 | AI助手、聊天机器人 |
| **约束式交互** | 在设计好的"按钮/模板/菜单"中让用户选择或填空 | 需要稳定输出的场景 |

**示例：简历助手**

**发散式交互：**
```
用户输入："帮我写一段有逻辑的简历介绍，我是学AI的，本科来自复旦，擅长产品经理方向。"
```

**约束式交互：**
```
用户填写：
- 姓名：[填写]
- 学历：[选择本科/硕士/博士]
- 专业：[填写]
- 亮点经历：[填写3句话]
然后点击【生成简历开头段落】
```

### 如何选择交互模式？

| 考虑因素 | 说明 |
|----------|------|
| **输出稳定性** | 需要稳定输出选约束式 |
| **用户自由度** | 需要灵活输入选发散式 |
| **AI能力** | AI能力强可采用发散式 |
| **业务场景** | 复杂流程选约束式，简单问答选发散式 |

## 五、AI产品经理工作流程

### 1. 需求分析阶段

```
1. 洞悉业务找到业务痛点
2. 明确AI能解决什么问题
3. 判断AI能力边界
4. 进行需求验证
```

### 2. 产品设计阶段

```
1. 功能设计
2. 交互模式选择（发散式 vs 约束式）
3. 输出PRD文档
4. 制定AI效果评估标准
```

### 3. 开发跟踪阶段

```
1. 协调算法、工程、设计团队
2. 跟踪开发进度
3. 效果评估
4. 调整优化
```

### 4. 上线迭代阶段

```
1. 数据分析
2. 用户反馈收集
3. 产品优化迭代
4. 持续监控AI效果
```

## 七、1个底座+N个业务应用

大模型时代的研发范式：

```
┌─────────────────────────────────────┐
│           N个业务应用                │
│  场景赋能层 + 个人应用层            │
└─────────────────┬───────────────────┘
                  │
┌─────────────────┴───────────────────┐
│           1个底座                  │
│  基础大模型 → 行业大模型 → 企业大模型│
└─────────────────────────────────────┘
```

### 产品经理的视角

| 层级 | 产品经理关注点 |
|------|---------------|
| **基础大模型** | 选择合适的底座模型 |
| **行业大模型** | 结合行业数据优化 |
| **企业大模型** | 企业私有化部署需求 |
| **业务应用** | 场景赋能、个人工具 |

## 八、AI时代对产品经理的要求

| 旧要求 | 新要求 |
|--------|--------|
| 写详细需求文档 | 用Prompt控制AI |
| 协调多方资源 | 理解AI能力边界 |
| 跟踪开发进度 | 评估AI效果 |
| 传统项目管理 | AI驱动的产品迭代 |

### 核心变化

```
传统产品经理 → AI产品经理
- 从"写规则"到"写Prompt"
- 从"协调开发"到"协调AI"
- 从"功能迭代"到"效果迭代"
```
