### 2.1 sizeof 和 内存对齐

#### 什么是内存对齐？

CPU读取内存时，希望数据"对齐"到特定边界，这样读取效率最高。

**对齐规则**：
- 每个成员的偏移量，必须是它自身大小的倍数
- 整个结构体的大小，必须是最大成员大小的倍数

#### 图解内存布局

```cpp
struct Test {
    char a;      // 1字节
    int b;       // 4字节
};
```

```
内存地址:  0    1    2    3    4    5    6    7
          ┌────┬────────┬────────┐
Test:     │ a  │ (空3字节)│   b   │
          └────┴────────┴────────┘
          char  凑对齐    int(4字节)

分析：
1. a 占 1 字节，放在地址 0
2. b 占 4 字节，要求偏移量是 4 的倍数
   → 跳过地址 1-3，放在地址 4-7
3. 总大小要是 4 的倍数 → 答案是 8 字节
```

#### 经典面试题

```cpp
// 面试题：这个结构体多大？
struct A {
    char a;     // 1字节
    int b;      // 4字节
    char c;     // 1字节
};
```

```
内存布局：
  0    1-3   4-7    8    9-15
┌────┬──────┬──────┬────┬─────────┐
│ a  │(补3) │  b   │ c  │ (补7)  │
└────┴──────┴──────┴────┴─────────┘
答案：16 字节
```

> **面试标准答案**：
> 答案是 8 字节（32位/64位可能不同，但原理一样）
> 分析：char占1字节，int占4字节要求对齐到4的倍数，所以跳到地址4-7

### 2.2 static 的作用

static 有 **3种用法**：

| 用法 | 代码示例 | 作用 |
|------|----------|------|
| 修饰局部变量 | `static int num = 0;` | 只初始化一次，值保留 |
| 修饰全局变量 | `static int g_count;` | 只能本文件使用 |
| 修饰函数 | `static void f();` | 只能本文件调用 |

#### 例子：static 局部变量

```cpp
void count() {
    static int num = 0;  // 只初始化一次！
    num++;
    printf("%d\n", num);
}

int main() {
    count();  // 输出 1
    count();  // 输出 2
    count();  // 输出 3
    return 0;
}
```

```
普通局部变量：存在栈上，每次调用都重新创建
static局部变量：存在静态数据区，程序开始就创建，程序结束才销毁
```

#### C++特有的 static

| 用法 | 代码示例 | 作用 |
|------|----------|------|
| 修饰成员变量 | `static int count;` | 所有对象共享一份 |
| 修饰成员函数 | `static void fun();` | 不用创建对象就能调用 |

> **面试标准答案**：static有3种作用：1.修饰局部变量，延长生命周期；2.修饰全局变量/函数，限制作用域；3.修饰类成员，所有对象共享

### 2.3 const 的作用

const 的4种写法：

```cpp
const int a = 10;     // a 是常量，不能修改
int const b = 20;     // 同上，两种写法等价

const int *p1;        // p1 指向的内容不能变
int * const p2;       // p2 指针本身不能变
const int * const p3; // 都不能变
```

#### 记忆方法

```cpp
// const 和谁近，谁就不能变

int const *p;     // const 在 * 前面 → "指向的内容"是常量
int * const p;     // const 在 p 前面 → "指针本身"是常量
```

### 2.4 malloc 和 new

#### 代码对比

```cpp
// C语言
int *p1 = (int*)malloc(sizeof(int));
free(p1);

// C++
int *p2 = new int;          // 分配
delete p2;                   // 释放

int *p3 = new int(100);     // 分配并初始化为100
delete p3;

int *arr = new int[10];     // 分配数组
delete[] arr;               // 释放数组要加[]
```

#### 详细对比

| 特性 | malloc | new |
|------|--------|-----|
| **类型** | C函数 | C++操作符 |
| **返回类型** | void*，需要强制转换 | 自动返回正确类型 |
| **构造函数** | 不调用 | 调用 |
| **析构函数** | 不调用 | 调用 |
| **失败处理** | 返回NULL | 抛异常(bad_alloc) |
| **内存初始化** | 不初始化（随机值） | 基本类型初始化为0 |

> **面试标准答案**：
> 5点区别：1.类型安全；2.构造函数；3.失败处理；4.内存初始化；5.底层实现

### 2.5 智能指针

#### 为什么要用智能指针？

```cpp
// 问题代码
void func() {
    Resource *r = new Resource();
    if (error) {
        return;  // 提前返回，内存泄漏！
    }
    delete r;
}

// 智能指针解决
void func() {
    unique_ptr<Resource> r = make_unique<Resource>();
    if (error) {
        return;  // 自动释放！
    }
}
```

#### 三种智能指针

```cpp
// 1. unique_ptr - 独占所有权
unique_ptr<int> p1 = make_unique<int>(10);
unique_ptr<int> p2 = move(p1);  // 可以移动，不能复制

// 2. shared_ptr - 共享所有权，有引用计数
shared_ptr<int> p3 = make_shared<int>(20);
shared_ptr<int> p4 = p3;  // 引用计数变成2

// 3. weak_ptr - 解决循环引用，不增加计数
weak_ptr<int> p5 = p3;
```

#### 循环引用问题

```cpp
// 问题：两个shared_ptr互相引用
class Parent {
public:
    shared_ptr<Child> child;  // 引用Child
};
class Child {
public:
    shared_ptr<Parent> parent;  // 引用Parent → 循环引用！
};
```

```
循环引用图解：
    ┌─────────┐          ┌─────────┐
    │ Parent  │────────→│  Child  │
    │ count=1 │←────────│ count=1 │
    └─────────┘          └─────────┘
         ↑                       │
         └───────────────────────┘
              互相引用，永远=1
```

#### 解决方法：weak_ptr

```cpp
class Child {
public:
    weak_ptr<Parent> parent;  // 改成 weak_ptr，不计数
};
```

> **面试标准答案**：
> 3种智能指针：unique_ptr独占、shared_ptr共享计数、weak_ptr不计数
> 循环引用：两个shared_ptr互相引用导致无法释放
> 解决：把一个改成weak_ptr

### 2.6 虚函数和多态

#### 什么是多态？

同一个函数调用，自动执行正确版本：

```cpp
class Animal {
public:
    virtual void speak() { }  // 定义"规矩"
};

class Dog : public Animal {
public:
    void speak() override {
        cout << "汪汪汪" << endl;
    }
};

class Cat : public Animal {
public:
    void speak() override {
        cout << "喵喵喵" << endl;
    }
};

void letSpeak(Animal* animal) {
    animal->speak();  // 自动调用正确的版本！
}
```

#### 为什么要用虚函数？

**因为父类在定义"规矩/接口"：**

```
Animal（父类）说：
    "我不管你是什么动物，但作为动物，你必须会叫"

Dog（狗）实现：   speak() → "汪汪汪"
Cat（猫）实现：   speak() → "喵喵喵"
```

#### 纯虚函数（强制子类实现）

```cpp
class Animal {
public:
    virtual void speak() = 0;  // =0 表示没有默认实现
};
// Animal 是抽象类，不能创建对象
```

#### 虚函数底层原理

```cpp
class Base {
public:
    virtual void func1();
    virtual void func2();
};

class Derived : public Base {
public:
    void func2() override;  // 重写func2
};
```

```
Base对象内存布局:              Base的虚表:
┌──────────────────────┐     ┌──────────────────┐
│  vfptr (虚表指针) ───┼────→│  Base::func1     │
│  a                   │     │  Base::func2     │
└──────────────────────┘     └──────────────────┘

Derived对象内存布局:           Derived的虚表:
┌──────────────────────┐     ┌──────────────────┐
│  vfptr (虚表指针) ───┼────→│  Base::func1     │
│  a                   │     │  Derived::func2  │ ← func2被覆盖
│  b                   │     └──────────────────┘
└──────────────────────┘

虚函数原理：运行时通过vfptr找虚表，再找函数地址 → 动态绑定
```

> **面试标准答案**：
> 1. 为了实现多态：不同对象调用同一函数，执行不同行为
> 2. 父类定义"规矩"：告诉子类必须实现这个功能
> 3. 底层：虚表+虚表指针，运行时动态绑定

### 2.7 虚析构函数

#### 问题场景

```cpp
class Base {
public:
    virtual ~Base() { }  // 虚析构
};

class Derived : public Base {
public:
    ~Derived() { }  // 自动变成虚析构
};

int main() {
    Base *p = new Derived();
    delete p;  // 现在会调用 Derived析构 → Base析构
}
```

```
虚析构函数前：只调用Base析构，Derived资源泄漏
虚析构函数后：先调用Derived析构 → 再调用Base析构，完整释放
```

> **面试标准答案**：
> 当用基类指针delete子类对象时，必须用虚析构函数
> 防止只调用基类析构导致子类资源泄漏

## 第三章 进程和线程

### 3.1 进程和线程的区别

#### 基本概念

```
进程（Process）：
    - 运行中的程序，是系统分配资源的基本单位
    - 有独立的地址空间（代码段、数据段、堆、栈）
    - 进程间隔离，一个进程崩溃不影响其他进程

线程（Thread）：
    - 进程内的执行单元，是CPU调度的基本单位
    - 共享进程的地址空间（代码段、数据段、堆）
    - 每个线程有独立的栈和寄存器
```

#### 图解对比

```
进程A的内存空间：
┌────────────────────────────────────────┐
│  代码段  │  数据段  │    堆     │  栈  │
│ (共享)   │ (共享)  │ (共享)    │(线程1)│
│                    │(共享)    │(线程2)│
└────────────────────────────────────────┘
          ↑           ↑          ↑
       线程共享     线程共享    线程独享

进程B的内存空间：
┌────────────────────────────────────────┐
│  代码段  │  数据段  │    堆     │  栈  │
└────────────────────────────────────────┘
         进程B自己用，和进程A完全隔离
```

#### 详细对比表

| 特性 | 进程 | 线程 |
|------|------|------|
| **资源拥有** | 独立地址空间 | 共享进程资源 |
| **开销** | 创建/销毁大 | 创建/销毁小 |
| **通信** | IPC较复杂 | 直接读写共享内存 |
| **切换** | 切换页表，慢 | 只切换寄存器和栈，快 |
| **隔离性** | 完全隔离 | 不隔离，可能冲突 |
| **崩溃影响** | 不影响其他进程 | 可能导致整个进程崩溃 |

#### 为什么需要线程？

```
场景：Web服务器处理1000个并发请求

用进程：创建1000个进程 → 开销巨大！
用线程：创建1000个线程 → 共享资源，开销小！
```

#### 线程 vs 进程的适用场景

| 场景 | 用进程 | 用线程 |
|------|--------|--------|
| 需要隔离 | ✅ 一个崩溃不影响其他 | ❌ 一个崩溃可能全崩 |
| 需要共享数据 | ❌ 复杂（需要IPC） | ✅ 直接读写 |
| 计算密集型 | ✅ | ❌ 线程切换开销 |
| I/O密集型 | ❌ | ✅ 线程开销小 |

> **面试标准答案**：
> 4点区别：1.资源拥有；2.开销；3.通信；4.隔离性
> 适用场景：需要隔离用进程，需要共享数据/高并发用线程

### 3.2 进程间通信方式

#### 5种通信方式一览

| 方式 | 说明 | 特点 |
|------|------|------|
| **管道** | 字节流，单向通信 | 简单，只能亲缘进程 |
| **命名管道(FIFO)** | 有名文件，可无关进程 | 解决匿名管道限制 |
| **消息队列** | 消息链表，有类型 | 可按类型读取 |
| **共享内存** | 同一块物理内存 | 最快，需同步 |
| **Socket** | 网络/本地都支持 | 最通用 |

#### 1. 管道（Pipe）

```
管道特点：
- 半双工（单向通信）
- 字节流（不是消息）
- 只能在亲缘进程间使用
```

```cpp
int fd[2];
pipe(fd);  // fd[0]读，fd[1]写

if (fork() == 0) {
    close(fd[1]);   // 子进程关闭写端
    char buf[100];
    read(fd[0], buf, sizeof(buf));
} else {
    close(fd[0]);   // 父进程关闭读端
    write(fd[1], "Hello", 5);
}
```

```
管道示意图：
    父进程                        子进程
       │                             │
       ├─→ [pipe缓冲区] ──────────→  │
       │   (内核缓冲区)              │
```

#### 2. 命名管道（FIFO）

```cpp
mkfifo("/tmp/myfifo", 0666);  // 创建命名管道

// 进程A写
int fd = open("/tmp/myfifo", O_WRONLY);
write(fd, "Hello", 5);

// 进程B读
int fd = open("/tmp/myfifo", O_RDONLY);
read(fd, buf, sizeof(buf));
```

#### 3. 消息队列

```cpp
int msgid = msgget(key, IPC_CREAT | 0666);

// 发送
struct msgbuf { long mtype; char mtext[100]; };
msgsnd(msgid, &buf, strlen(buf.mtext), 0);

// 接收
msgrcv(msgid, &buf, sizeof(buf.mtext), 1, 0);
```

#### 4. 共享内存

```
共享内存是最快的IPC方式！不需要内核中转
```

```cpp
int shmid = shmget(key, 1024, IPC_CREAT | 0666);
char *shmaddr = (char*)shmat(shmid, NULL, 0);  // 附加
strcpy(shmaddr, "Hello");
shmdt(shmaddr);  // 分离
shmctl(shmid, IPC_RMID, NULL);  // 删除
```

```
共享内存示意图：
    进程A                    进程B
       │                        │
       ├────→ [物理内存] ←──────┤
       │      (同一块)          │
    直接读写，不用内核中转，最快！
```

#### 5. Socket

```cpp
// 本地Socket（AF_UNIX）
int sockfd = socket(AF_UNIX, SOCK_STREAM, 0);
// 绑定、监听、连接...
```

#### 通信方式对比

| 方式 | 速度 | 持久性 | 需要同步 |
|------|------|--------|----------|
| 管道 | 快 | 随进程 | 需同步 |
| FIFO | 快 | 随内核 | 需同步 |
| 消息队列 | 中 | 随内核 | 需同步 |
| 共享内存 | 最快 | 随内核 | **必须同步** |
| Socket | 慢 | 可持久 | 需同步 |

> **面试标准答案**：
> 5种方式：管道、消息队列、共享内存、信号量、Socket
> 速度排名：共享内存 > 管道/FIFO > 消息队列 > Socket

### 3.3 线程同步方式

#### 为什么要同步？

```cpp
// 问题：两个线程同时修改同一个变量
int count = 0;

void* worker(void* arg) {
    for (int i = 0; i < 100000; i++) {
        count++;  // 不是原子操作！
    }
}
// 结果不确定！可能是100000~200000
```

```
count++ 实际是3条指令：
1. 读取 count 到寄存器
2. 寄存器+1
3. 写回 count
```

#### 同步方式一览

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **互斥锁** | 保证同时只有一个线程访问 | 保护共享变量 |
| **读写锁** | 读多写少场景 | 高读低写 |
| **条件变量** | 等待某个条件成立 | 生产者消费者 |
| **信号量** | 控制资源数量 | 限流、资源池 |

#### 1. 互斥锁（Mutex）

```cpp

### 3.4 线程池

#### 为什么要用线程池？

```
不用线程池：每来一个任务，创建新线程 → 开销大
用线程池：预先创建固定线程，任务入队列 → 复用
```

#### 线程池原理

```
线程池结构：
┌─────────────────────────────────────────┐
│              任务队列                    │
│    [task1] [task2] [task3] [task4] ... │
└──────────┬──────────────────────────────┘
           │
    ┌──────┴──────┬───────┬───────┐
    │             │       │       │
┌───┴───┐    ┌───┴───┐ ┌───┴───┐ ┌───┴───┐
│线程1  │    │线程2  │ │线程3  │ │线程4  │
└───────┘    └───────┘ └───────┘ └───────┘
    │             │       │       │
    └──────┬──────┴───────┴───────┘
           │
    线程不断从队列取任务执行
```

#### 线程池的优点

| 优点 | 说明 |
|------|------|
| **复用线程** | 避免频繁创建/销毁的开销 |
| **控制并发** | 限制最大线程数，防止系统过载 |
| **提高响应** | 任务来了马上就能执行 |

> **面试标准答案**：
> 线程池：预先创建固定数量线程，任务入队列，线程取任务执行
> 优点：复用线程、控制并发、提高响应速度

### 4.2 死锁条件与处理

#### 什么是死锁？

```
死锁：两个或多个线程互相等待对方释放资源，谁都无法继续执行

例子：
    线程A：持有锁1，等待锁2
    线程B：持有锁2，等待锁1
    → 死锁！
```

#### 死锁的4个必要条件
死锁：多个进程互相等待对方持有的资源，导致都无法推进

产生条件（四个都满足才会死锁）：
1. 互斥：资源只能一个进程用
2. 占有并等待：拿着资源还想要更多
3. 不可抢占：资源不能被强制拿走
4. 循环等待：形成等待环路

解决：
- 预防：打破四个条件之一
- 避免：银行家算法
- 检测：资源分配图
- 解除：剥夺资源/杀死进程

| 条件 | 说明 |
|------|------|
| **互斥** | 资源只能被一个线程持有 |
| **持有并等待** | 持有资源，还在等待其他资源 |
| **不可抢占** | 资源不能被强制释放 |
| **循环等待** | 形成线程间的循环等待链 |

```
死锁形成图示：
    线程A ──持有1──→ 等待2 ──→ 线程B ──持有2──→ 等待1 ──→ 线程A
         ←───────────────────────────── 形成循环 ←──────────
```

#### 避免死锁：按固定顺序加锁

```cpp
// 错误：不同线程按不同顺序加锁 → 可能死锁
void* threadA(void*) { lock(m1); lock(m2); }
void* threadB(void*) { lock(m2); lock(m1); }  // 可能死锁！

// 正确：永远按固定顺序加锁 → 不会死锁
void* threadA(void*) { lock(m1); lock(m2); }
void* threadB(void*) { lock(m1); lock(m2); }  // 不会死锁！
```

> **面试标准答案**：
> 死锁4个条件：互斥、持有并等待、不可抢占、循环等待
> 避免方法：按固定顺序加锁

### 7.2 CUDA编程模型

#### 线程层次结构

```
Grid（网格）
│
├── Block 0 (0,0,0)     ├── Block 1 (1,0,0)     ├── Block 2 (2,0,0)
│   ├── Thread 0-255       ├── Thread 0-255       ├── Thread 0-255
│   └── 共享内存           └── 共享内存           └── 共享内存
│
└── Shared Memory ← 同一Block的线程共享
    │
    └── Registers ← 每个线程私有
```

```cpp
// kernel函数：在GPU上并行执行
__global__ void vectorAdd(int *a, int *b, int *c, int n) {
    // 计算全局线程ID
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

int main() {
    // 启动配置：256个线程/块，n/256个块
    int threadsPerBlock = 256;
    int blocksPerGrid = (n + threadsPerBlock - 1) / threadsPerGrid;

    vectorAdd<<<blocksPerGrid, threadsPerBlock>>>(d_a, d_b, d_c, n);
}
```

#### 关键概念

| 概念 | 说明 |
|------|------|
| **blockIdx** | 线程块在网格中的索引 |
| **threadIdx** | 线程在块中的索引 |
| **blockDim** | 每个块的线程数 |
| **gridDim** | 网格中块的数量 |
| **warp** | 32个线程为一组，SIMD执行 |

### 7.3 CUDA内存层级

```
GPU全局内存（显存）
│  容量大（GB级），延迟高（400-800周期）
│  所有线程可访问
│
└──→ 共享内存（Shared Memory）
     容量小（KB级），延迟低（1周期）
     同Block线程共享
     │
     └──→ 寄存器
          速度最快，线程私有
```

| 内存类型 | 容量 | 延迟 | 共享范围 |
|----------|------|------|----------|
| 全局内存 | GB级 | 400-800周期 | 所有线程 |
| 共享内存 | KB级 | 1周期 | 同Block |
| 寄存器 | - | 最快 | 线程私有 |
| 常量内存 | MB级 | 1周期（有缓存） | 所有线程 |

### 7.4 并行规约（Parallel Reduction）

```cpp
__global__ void reduce(int *data, int *result) {
    __shared__ int s[256];  // 共享内存

    int tid = threadIdx.x;
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    s[tid] = data[idx];
    __syncthreads();

    // 两两合并
    for (int s = blockDim.x/2; s > 0; s >>= 1) {
        if (tid < s) {
            s[tid] += s[tid + s];
        }
        __syncthreads();
    }

    if (tid == 0) {
        result[blockIdx.x] = s[0];
    }
}
```

```
规约过程（8个元素）：
Step 1: [a0,a1,a2,a3,a4,a5,a6,a7]
         ↓  ↓  ↓  ↓  ↓  ↓  ↓  ↓
Step 2: [a0+a1, a2+a3, a4+a5, a6+a7]
             ↓          ↓
Step 3: [a0+a1+a2+a3, a4+a5+a6+a7]
                 ↓
Step 4: [总和] ✓
```

### 7.5 GEMM矩阵乘法优化（面试高频）

#### 朴素矩阵乘法

```cpp
__global__ void gemmNaive(float *A, float *B, float *C, int N) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;

    float sum = 0.0f;
    for (int k = 0; k < N; k++) {
        sum += A[row * N + k] * B[k * N + col];
    }
    C[row * N + col] = sum;
}
```

#### 优化的矩阵乘法（分块 + 共享内存）

```cpp
#define TILE_SIZE 16

__global__ void gemmTiled(float *A, float *B, float *C, int N) {
    __shared__ float As[TILE_SIZE][TILE_SIZE];
    __shared__ float Bs[TILE_SIZE][TILE_SIZE];

    int row = blockIdx.y * TILE_SIZE + threadIdx.y;
    int col = blockIdx.x * TILE_SIZE + threadIdx.x;
    float sum = 0.0f;

    for (int m = 0; m < N/TILE_SIZE; m++) {
        // 加载tile到共享内存
        As[threadIdx.y][threadIdx.x] = A[row * N + m * TILE_SIZE + threadIdx.x];
        Bs[threadIdx.y][threadIdx.x] = B[(m * TILE_SIZE + threadIdx.y) * N + col];
        __syncthreads();

        // 计算这个tile
        for (int k = 0; k < TILE_SIZE; k++) {
            sum += As[threadIdx.y][k] * Bs[k][threadIdx.x];
        }
        __syncthreads();
    }

    C[row * N + col] = sum;
}
```

#### CUDA面试标准答案

> **Q：CUDA的线程层次是怎样的？**
> ```
> Grid → Block → Thread
> - Grid：整个GPU，所有线程块组成
> - Block：线程块，同一Block线程可共享Shared Memory
> - Thread：最小执行单元
> - Warp：32个线程为一组，SIMD执行
> ```

> **Q：如何计算线程的全局ID？**
> ```
> int globalId = blockIdx.x * blockDim.x + threadIdx.x;
> ```

> **Q：Shared Memory和Global Memory的区别？**
> ```
> Global Memory：容量大（GB级），延迟高（400-800周期），所有线程可访问
> Shared Memory：容量小（KB级），延迟低（1周期），同Block线程共享
> ```

> **Q：什么是coalescing访问？为什么重要？**
> ```
> Coalescing访问是指同一warp的线程访问连续的内存地址。
> 优点：多个线程的访问可以合并为一次内存事务，大幅提升带宽利用率。
> ```

### 9.3 规划（Planning）

#### ReAct模式（最常用）

```python
# ReAct = Reasoning + Acting
"""
ReAct的核心思想：
1. Thought（思考）：分析当前情况，决定下一步
2. Action（行动）：执行某个动作
3. Observation（观察）：获取行动结果
4. 重复直到任务完成
"""

def react_loop(query, agent):
    memory = []  # 记忆
    for _ in range(max_iterations):
        # 1. 思考
        thought = agent.think(query, memory)

        # 2. 决定行动
        action = agent.decide_action(thought)

        # 3. 执行行动
        result = agent.execute(action)

        # 4. 观察结果
        observation = agent.observe(result)
        memory.append((thought, action, observation))

        # 5. 检查是否完成
        if agent.is_finished(query, memory):
            break

    return agent.generate_response(memory)
```

```
ReAct流程：
    ┌─────────────┐
    │   Query     │
    └──────┬──────┘
           ↓
    ┌─────────────┐
    │   Thought   │ ← 分析情况
    └──────┬──────┘
           ↓
    ┌─────────────┐
    │   Action    │ ← 决定做什么
    └──────┬──────┘
           ↓
    ┌─────────────┐
    │ Execution   │ ← 执行
    └──────┬──────┘
           ↓
    ┌─────────────┐
    │Observation  │ ← 观察结果
    └──────┬──────┘
           ↓
        ┌──┴──┐
        ↓     ↑
      继续   完成
```

### 9.4 记忆（Memory）

#### 记忆类型

| 类型 | 说明 | 实现 |
|------|------|------|
| **短期记忆** | 当前对话上下文 | Token窗口 |
| **长期记忆** | 历史积累的知识 | 向量数据库 |
| **工作记忆** | 当前任务相关信息 | 临时缓存 |

```python
# 短期记忆：滑动窗口
class ShortTermMemory:
    def __init__(self, max_tokens=4096):
        self.messages = []
        self.max_tokens = max_tokens

    def add(self, msg):
        self.messages.append(msg)
        while self.count_tokens() > self.max_tokens:
            self.messages.pop(0)  # 超出就裁剪

# 长期记忆：向量存储
class LongTermMemory:
    def __init__(self):
        self.vector_db = FAISS()
        self.embedder = OpenAIEmbedder()

    def store(self, text):
        embedding = self.embedder.embed(text)
        self.vector_db.add(embedding, text)

    def retrieve(self, query, top_k=5):
        query_emb = self.embedder.embed(query)
        return self.vector_db.search(query_emb, top_k)
```

## 附录三：核心概念大白话解释

### 1. CUDA是什么？

```
CUDA 不是编程语言！是 NVIDIA 的 GPU编程平台

类比：
- 编程语言：C/C++、Python
- CUDA：NVIDIA给GPU编程用的工具 + C语言扩展

写CUDA代码：
- 用 C语言 + CUDA语法 写程序
- 编译后能在 GPU 上运行
```

```
CPU编程：写代码 → 编译 → CPU运行

GPU编程：写代码(C+CUDA扩展) → nvcc编译 → GPU运行
                       ↑
                 CUDA就是干这个
```

### 2. Kubernetes是什么？

```
K8s = 装应用的"箱子"管理系统

场景：你App要部署到10台服务器
- 传统：手动一个个配置，累死
- K8s：打包成容器，一键部署，自动管理
```

```
K8s核心概念：
- Pod：最小的部署单元（装1个或多个容器）
- Service：给Pod提供稳定的访问入口+负载均衡
- Deployment：声明"我要3个Pod运行"，K8s自动管理
```

### 3. 你的FitGenius用LangChain了吗？

回答：没有直接用，但实现了类似的思想！

LangChain帮你做：

你的App手写：
