**八股速记**：

**Q: 进程和线程的区别？**

```
进程：资源分配的基本单位，有独立地址空间
线程：CPU调度的基本单位，共享进程资源

区别：
1. 进程有独立地址空间，线程共享
2. 进程切换开销大，线程小
3. 线程更轻量，创建/销毁更快
4. 进程间通信复杂，线程可以直接共享数据
```

**Q: 什么是死锁？**

```
死锁：多个进程互相等待对方持有的资源，导致都无法推进

产生条件（四个都满足才会死锁）：
1. 互斥：资源只能一个进程用
2. 占有并等待：拿着资源还想要更多
3. 不可抢占：资源不能被强制拿走
4. 循环等待：形成等待环路

解决：
- 预防：打破四个条件之一
- 避免：银行家算法
- 检测：资源分配图
- 解除：剥夺资源/杀死进程
```

**银行家算法（详细讲解）**

```
为什么叫银行家算法？
→ 因为像银行放贷：先判断能不能收回来，再决定放不放贷

核心思想：资源分配前，先判断"分配后系统是否安全"

数据结构：
┌────────────┬────────────┬────────────┐
│   进程     │  Max需求    │ Allocation已分配│
├────────────┼────────────┼────────────┤
│    P0      │   10       │     5      │
│    P1      │    4       │     2      │
│    P2      │    9       │     2      │
└────────────┴────────────┴────────────┘

Available = 10 表示系统当前可用资源数

Need = Max - Allocation 表示还需要的资源数

       Need
P0: 10 - 5 = 5
P1: 4 - 2 = 2
P2: 9 - 2 = 7

安全序列例子：
1. 先给P1（Need=2最小）
   P1完成，释放资源 → Available = 10 + 2 = 12

2. 再给P0（Need=5）
   P0完成，释放资源 → Available = 12 + 5 = 17

3. 再给P2（Need=7）
   P2完成，释放资源 → Available = 17 + 7 = 24

执行顺序：P1 → P0 → P2
每个进程都能完成，所以是"安全状态"

判断是否安全：
Available >= Need(P1)? → 10 >= 2 ✓
Available >= Need(P0)? → 12 >= 5 ✓
Available >= Need(P2)? → 17 >= 7 ✓

如果找不到这样的序列 → 不安全，可能死锁
```

**资源分配图（画图讲解）**

```
两种元素：
┌─────┐         资源类型：R1, R2, R3...
│ P1  │         资源实例：用圆点表示数量
└─────┘         进程：P1, P2, P3...

两种边：
申请边（Process → Resource）：进程在请求这个资源
分配边（Resource → Process）：资源已经分配给这个进程

图形表示：
    P1 --申请--> R1
    ↑              ↓
    P2 <--分配-- R2

    图中：
    R1有1个实例，已分配给P2，P1在申请R1
    R2有1个实例，已分配给P1，P2在申请R2

死锁判断：
1. 图中存在环（环路）
2. 环路中每类资源只有一个实例
→ 一定是死锁

如果资源有多个实例：
环路存在 ≠ 一定死锁
需要进一步分析

举例（单个资源实例，有环=死锁）：
    P1 → R1 → P2 → R2 → P1
    P1持有R2申请R1
    P2持有R1申请R2
    互相等待 → 死锁
```

**Q: 什么是虚拟内存？**

```
虚拟内存：用硬盘空间模拟内存，让程序可以使用超出物理内存的空间

为什么需要：
- 程序越来越多，但物理内存有限
- 每个程序都想要连续的地址空间
- 虚拟内存让程序以为自己有独立的连续大内存

核心概念：
- 虚拟地址：程序看到的地址
- 物理地址：实际内存条的地址
- MMU：Memory Management Unit，专门做地址转换的硬件
```

**MMU是什么？**

```
MMU = 内存管理单元，CPU里的一块硬件

工作流程：
程序(虚拟地址) → MMU(查页表) → 物理内存(物理地址)

举例：
虚拟地址 0x1000 → MMU转换 → 物理地址 0x3A12000
```

**地址怎么转换？（分页机制）**

```
虚拟地址结构：  [页号(Page Number) | 页内偏移(Page Offset)]
                ↓
              查页表
                ↓
物理地址结构：  [帧号(Frame Number) | 页内偏移]

假设：页大小 = 4KB = 4096字节

虚拟地址 0x1000 转物理地址：
- 0x1000 / 4096 = 1  → 页号 = 1
- 0x1000 % 4096 = 0  → 页内偏移 = 0

查页表：页号1 → 物理帧号3
物理地址 = 3 × 4096 + 0 = 0x3000
```

**页面置换算法（当内存不够时）**

```
当需要的页不在内存时，就要换出来

常见算法：
1. FIFO：先进先出，最早进来的先换出去（简单但可能换出常用页）
2. LRU：Least Recently Used，最近最少使用的换出去（近似最优，但实现复杂）
3. LFU：Least Frequently Used，使用频率最低的换出去
4. OPT：最优置换，理论上最优但无法实现（用于对比）

例子理解：
假设只能用3个页框，访问顺序：1,2,3,4,1,2,5,1,2,3,4,5

FIFO结果：缺页9次
LRU结果：缺页10次
（不同算法效果不同，实际LRU更优）
```

**TLB是什么？（加速机制）**

```
TLB = Translation Lookaside Buffer

因为每次地址转换都要查页表，页表又在内存里
→ 每次转换要两次内存访问（查页表 + 访问数据）
→ 太慢了！

TLB是MMU里的一块高速缓存，存放最近用过的页表项
→ 命中TLB就直接转换，不用查内存
→ 命中率约90%，大幅加速
```

**虚拟内存带来的好处**

```
1. 程序可以使用比物理内存更大的空间
2. 内存隔离，一个程序崩溃不影响其他程序
3. 内存共享，多个程序可以共用相同的物理页（如动态库）
4. 按需分配，不用一次把程序全部加载到内存
```

**缺页中断（Page Fault）**

```
程序需要的页不在内存时，触发缺页中断

流程：
1. CPU访问某个虚拟地址
2. MMU查页表，发现页不在内存
3. 触发缺页中断（Page Fault）
4. OS把需要的页从硬盘加载到内存
5. 更新页表
6. 重新执行刚才的指令

如果内存满了，还需要先换出一页（页面置换）
```

**八股速记**：

**Q: 什么是索引？**

```
索引：加速查询的数据结构，类似书的目录

原理：
- B+树是最常用的
- O(log n) 查找复杂度
- 但会占用额外空间

什么时候建索引：
- WHERE条件经常查的字段
- 主键自动建索引
- 外键建索引

什么时候不要建：
- 数据量小
- 频繁更新的字段
- 重复值多的字段
```

**Q: 事务的ACID是什么？**

```
A - Atomicity（原子性）：
  要么全做，要么全不做
  一笔转账：扣钱+加钱，必须同时成功或同时失败

C - Consistency（一致性）：
  事务前后数据都是合法的状态
  转账前后，总金额必须不变

I - Isolation（隔离性）：
  并发事务互不干扰
  两个转账同时操作同一账户，结果要正确

D - Durability（持久性）：
  提交后数据永久保存
  提交成功就意味着数据一定写入磁盘
```

**事务的详细解释**

```
为什么需要事务？
→ 因为操作可能失败（比如断电、网络中断）
→ 没有事务，数据可能处于"半完成"状态

举例：ATM转账1000元
步骤1：账户A减1000
步骤2：账户B加1000

如果步骤1成功后断电了？
- 没有事务：账户A少了1000，B没加钱，钱消失了！
- 有事务：自动回滚，两边都不变
```

**并发问题（隔离性详解）**

```
多个事务同时执行时，会出现这些问题：

1. 脏读（Dirty Read）：
   事务A还没提交，事务B就读到了A修改的数据
   → 如果A回滚，B读到的就是"脏"数据

2. 不可重复读（Non-repeatable Read）：
   事务A两次读取同一行数据，结果不一样
   → 因为事务B在A两次读取之间修改并提交了

3. 幻读（Phantom Read）：
   事务A按条件查询两次，得到的结果数量不一样
   → 因为事务B在A两次查询之间插入了新数据

4. 丢失更新（Lost Update）：
   事务A和B同时修改同一数据，后提交的会覆盖先提交的
```

**隔离级别（从低到高）**

```
1. Read Uncommitted（读未提交）：
   可能脏读，但能看到未提交的数据

2. Read Committed（读已提交）：
   只能看到已提交的数据，但不可重复读

3. Repeatable Read（可重复读）：
   同一事务中多次读取结果相同，但可能幻读
   （MySQL默认就是这个）

4. Serializable（串行化）：
   完全串行执行，彻底安全，但性能最差

隔离级别越高 → 数据越安全 → 性能越差
```

**MVCC（多版本并发控制）**

```
MVCC = Multi-Version Concurrency Control

原理：每个事务看到的是"快照"（特定时间点的数据）
- 不同事务可以看到不同版本的数据
- 写操作不阻塞读操作
- 大幅提高并发性能

MySQL的InnoDB就是用MVCC实现可重复读的
```

**事务的状态**

```
活动状态 → 部分提交状态 → 提交状态
    ↓            ↓            ↓
  执行中      碰巧先写入     成功完成
             日志但还没
             真正提交

活动状态 → 中止状态
    ↓
  失败/回滚
```

**回滚日志（Undo Log）**

```
事务回滚时，靠Undo Log恢复数据

例子：把 x 从 5 改成 10
1. 先记录Undo：x=5
2. 修改x=10
3. 如果回滚，用Undo恢复x=5

Undo Log特点：
- 写入顺序和redo相反
- 用于事务回滚和MVCC快照读取
```

**Q: TCP三次握手？（必须画出来）**

```
状态说明：
- LISTEN：服务器在监听端口，等待连接
- SYN_SENT：客户端发出SYN，等待确认
- SYN_RCVD：服务器收到SYN，发送SYN+ACK，等待确认
- ESTABLISHED：连接建立成功，双方可以传输数据

三次握手详细过程：

Step 1：客户端 → 服务端（SYN=1, seq=x）
┌─────────────────────────────────────┐
│ 客户端：我要连接你，我的起始序列号是x │
└─────────────────────────────────────┘
           ↓
Step 2：服务端 → 客户端（SYN=1, ACK=1, seq=y, ack=x+1）
┌─────────────────────────────────────────────┐
│ 服务端：收到！我同意连接，我的起始序列号是y   │
│       ack=x+1 表示"我收到了x之前的字节"     │
└─────────────────────────────────────────────┘
           ↓
Step 3：客户端 → 服务端（ACK=1, seq=x+1, ack=y+1）
┌─────────────────────────────────────────────┐
│ 客户端：收到！连接建立完成，双方开始传数据    │
└─────────────────────────────────────────────┘

为什么需要三次？（核心问题）

→ 验证"双向通信"能力

第1次：客户端证明自己能发
第2次：服务端证明自己能收 + 能发
第3次：客户端证明服务端能收

如果是两次握手：
- 客户端发SYN，服务端回ACK
- 但服务端不知道自己发的ACK客户端能不能收到
- 可能客户端根本没收到，服务端就认为建好了

四次行不行？行，但没必要，第三次就够了
```

**Q: TCP四次挥手？（断开连接）**

```
关闭原则：谁先发起close，谁就进入TIME_WAIT

详细过程（客户端先发起）：

Step 1：客户端 → 服务端（FIN=1, seq=u）
┌─────────────────────────────────────┐
│ 客户端：我数据发完了，想断开连接     │
└─────────────────────────────────────┘
           ↓ 进入 FIN_WAIT_1 状态
Step 2：服务端 → 客户端（ACK=1, ack=u+1）
┌─────────────────────────────────────┐
│ 服务端：收到，我看看还有没有数据要发  │
└─────────────────────────────────────┘
           ↓ 进入 CLOSE_WAIT 状态
           （客户端收到ACK后进入 FIN_WAIT_2）
           ↓
Step 3：服务端 → 客户端（FIN=1, seq=v）
┌─────────────────────────────────────┐
│ 服务端：我也发完了，断开连接         │
└─────────────────────────────────────┘
           ↓ 进入 LAST_ACK 状态
Step 4：客户端 → 服务端（ACK=1, ack=v+1）
┌─────────────────────────────────────┐
│ 客户端：好的，断开                   │
└─────────────────────────────────────┘
           ↓
    客户端等2MSL（最大报文生存时间）后才真正关闭
    服务端收到ACK后立即关闭

为什么等待2MSL？
→ 确保服务端收到ACK，如果服务端没收到会重发FIN
→ 让本连接的报文都从网络中消失，防止影响新连接
```

**Q: TCP和UDP的区别？**

```
TCP（ Transmission Control Protocol）：
- 面向连接：先建立连接再传输数据
- 可靠传输：有确认、重传、排序机制
- 拥塞控制：防止网络过载
- 速度较慢，但保证正确性
- 全双工：双方可以同时收发

UDP（User Datagram Protocol）：
- 无连接：直接发送数据
- 不可靠：不保证送达、不保证顺序
- 没有拥塞控制
- 速度快，但可能丢包
- 简单高效

对比表格：
| 特性     | TCP        | UDP          |
|----------|------------|--------------|
| 连接     | 有连接      | 无连接        |
| 可靠性   | 可靠        | 不可靠        |
| 速度     | 慢          | 快            |
| 用途     | 网页、邮件   | 视频、直播、游戏 |
| 首部大小  | 20-60字节   | 8字节         |
```

**Q: HTTP和HTTPS的区别？**

```
HTTP（HyperText Transfer Protocol）：
- 明文传输，数据不加密
- 端口：80
- 不安全，但速度快

HTTPS（HTTP Secure）：
- HTTP + SSL/TLS 加密
- 端口：443
- 加密传输，数据安全
- 需要证书，有一定性能开销

HTTPS的加密过程（握手）：
1. 客户端hello（支持的加密算法）
2. 服务器hello（选中的算法 + 证书）
3. 客户端验证证书，生成随机数，用公钥加密
4. 服务器用私钥解密，双方用随机数生成对称密钥
5. 之后的通信用对称密钥加密（快）
```

**DNS解析过程（递归+迭代）**

```
用户输入 www.example.com

1. 浏览器缓存 → 查本机 hosts → 查本机DNS缓存
2. 没有的话，问本地DNS服务器（通常运营商提供）
3. 本地DNS服务器：
   - 先问根域名服务器（全球13个）
   - 根域名告诉它：.com的域名服务器地址
   - 再问.com域名服务器
   .com域名服务器告诉它：example.com的DNS服务器
   再问example.com的DNS服务器
   终于得到 www.example.com 的IP地址
4. 本地DNS缓存这个结果，返回给浏览器

整个过程：客户端到本地DNS（递归）
         本地DNS一步步问（迭代）
```

**浏览器输入URL到显示页面的过程**

```
1. URL解析：分离出协议、域名、路径、端口
2. DNS查询：域名 → IP地址
3. 建立TCP连接：三次握手
4. 发起HTTP请求：GET /index.html HTTP/1.1
                 Host: www.example.com
5. 服务器处理请求：查路由、执行业务逻辑、查询数据库
6. 服务器返回响应：HTTP/1.1 200 OK
                   Content-Type: text/html
                   [HTML内容]
7. 浏览器解析HTML：构建DOM树
8. 遇到CSS/JS：发起新请求获取资源
9. 浏览器渲染页面：CSS + DOM = 渲染树 → 布局 → 绘制
10. TCP四次挥手：断开连接
```
